## Codey Rocky USB Serial Communication Specifications
The low-level serial infrastructure of Codey Rocky is managed by a customized MicroPython interpreter running on an ESP32 chip. To protect its core communication pipeline and support real-time block editing, Makeblock routes the physical USB-UART connection through a high-priority C++ state-machine filter.
This architectural design creates an asymmetric behavior between serial transmission and reception.
------------------------------
## 1. Serial Output (TX)
Serial output is fully open and straightforward to use.

* Mechanism: Using the standard Python print() function or sys.stdout.write() outputs plain-text strings down the USB cable.
* Capture: Any standard terminal emulator can capture these strings directly from the virtual COM port.
* Terminal Settings:
* Baud Rate: 115200
   * Data Bits: 8
   * Parity: None
   * Stop Bits: 1
   * Flow Control: None (Disabling RTS/CTS is required to prevent data freezing).

------------------------------
## 2. Serial Input (RX) — The Roadblock
Standard ASCII user text inputs (such as typing commands into PuTTY or Haiku Serial Connector) are completely blocked and ignored at the firmware level.

* The Gatekeeper: The background C++ firmware intercepts the incoming UART data stream before it can reach the MicroPython abstraction layer.
* The Filter: The firmware continuously scans incoming bytes for Makeblock's proprietary binary packet signatures.
* The Purge: If an incoming stream fails to conform to the strict protocol framing structure, the firmware flags the data as corrupted transmission noise and purges it from the buffer. This prevents custom text strings from reaching sys.stdin.
* Stripped Hardware Hooks: Makeblock stripped the standard MicroPython machine module from the production firmware. Users cannot initialize a raw hardware UART(0) object to manually hijack the pins or bypass the filter.

------------------------------
## 3. The Proprietary F3 01 Protocol Specification
To pass data successfully through the USB serial RX port, inputs must be formatted as raw hexadecimal strings that precisely mirror the "F3 & F4 Protocol Group" syntax:

[Header]      [Cmd ID]    [Data Length]    [Payload / Sub-IDs]    [Checksum]    [Footer]
 0xF3 0x01      0x0D         0x00 0x07        ... bytes ...          0xXX         0xF4

## Packet Breakdowns:

* Header (0xF3 0x01): The rigid magic number required to open the software gate.
* Command ID:
* 0x0E: System Info/Query mode (e.g., returns firmware version text strings like 23.01.017.006).
   * 0x0D: Live execution mode. Routes subsequent payload bytes directly to internal hardware driver routines.
* Data Length (2 Bytes): A 16-bit integer declaring exactly how many payload bytes follow.
* Payload Sub-IDs (0x0D Mode): Maps to physical hardware actuators:
* 0x01: LED Matrix Display
   * 0x02: RGB LED Indicator
   * 0x03: Speaker / Buzzer
   * 0x04: Rocky Chassis Motors
* Checksum: A single-byte custom longitudinal redundancy check (LRC) or basic XOR checksum generated by hashing the payload bytes.
* Footer (0xF4): Validates the end of the frame transmission.

⚠️ Script Interruption: Sending valid mBlock live handshake packets down the USB wire triggers a hardware-level interrupt. This instantly kills any locally running MicroPython script and forces Codey into an empty "Live Monitor Listening Mode."

------------------------------
## 4. Alternative Input Methods
Because the USB serial input path is locked down by the protocol filter, alternative methods must be used to pass dynamic text data (such as Wi-Fi credentials):

| Interface | Method Type | MicroPython Implementation Mechanism |
|---|---|---|
| Infrared (IR) | Optical UART | codey.ir.receive() accepts raw asynchronous serial strings ending in \n via an external IR transmitter (like an ESP32 or a Flipper Zero raw capture player). |
| Local Flash | JSON Configuration File | Read variables on boot from a local file via uos and ujson. Update credentials by running a temporary write script inside the mBlock IDE. |
| Wi-Fi / Network | TCP / UDP Sockets | Hardcode a base profile once, then use the standard usocket module to open an open text communication port (e.g., port 8080) over the local network. |
| 4-Pin Module Port | I2C Slave Emulation | Connect an external microcontroller (like an Arduino) to the mBuild side-port. Force it to mimic a Makeblock expansion device address to feed variables into the neuron module loop. |
